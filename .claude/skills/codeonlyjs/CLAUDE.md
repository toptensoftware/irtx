Purpose & context
Brad is building "Hitch," a minimal web-based email client named after Christopher Hitchens, designed as a personal side project to improve upon existing alternatives like RoundCube and SnappyMail. The primary differentiator is cross-mailbox conversation threading, which addresses limitations Brad identified in current solutions. This builds on "imaptest," a working proof-of-concept Brad created years ago that successfully implemented backend conversation threading with Vue.js frontend but was read-only. Hitch represents a complete rebuild incorporating lessons learned, modern tech preferences, and adding missing write operations.
The project follows a deliberately opinionated and minimal philosophy, limiting scope to Brad's specific needs: markdown email composition, simple folder structure (inbox/sent/archive/junk), search-focused workflow, and essential keyboard shortcuts for quick actions like archive, delete, and reply. The architecture emphasizes progressive enhancement where basic IMAP client functionality works immediately while conversation threading comes online gradually as background indexing progresses.
Current state
Brad is actively developing the core IMAP functionality and user interface components. Recent work includes implementing efficient message sorting and pagination using IMAP SORT extensions, designing clean URL schemes for bookmarkable email navigation, and building responsive HTML layouts with resizable panels. He's successfully resolved IMAP connection issues with docker-mailserver and confirmed excellent server capability support including QRESYNC, CONDSTORE, IDLE, and THREAD=REFERENCES.
Current technical implementations include session-based authentication using IMAP credentials directly, secure password storage with AES-256-GCM encryption, and message list rendering with features like attachment detection, draft identification, and smart avatar systems using domain favicons with Gravatar fallbacks. The frontend uses CSS Grid for optimal performance in message list layouts with proper text ellipsis handling.

Key learnings & principles
Brad has established that separation of concerns enables developing and testing core email client functionality independently from threading features, allowing incremental value delivery while maintaining robustness. The progressive enhancement approach ensures basic functionality works immediately while advanced features like conversation threading appear gradually as background processing completes.
Key technical insights include using Message-IDs rather than IMAP UIDs for permanent, cross-server stable message identification in URLs, implementing separate connections for IMAP operations versus IDLE monitoring due to protocol limitations, and leveraging server-side capabilities like Dovecot's PREVIEW extension to offload text extraction work rather than requiring client-side HTML/MIME parsing.
Security considerations emphasize session-scoped password validation and timing-safe password comparison to prevent unauthorized access and timing attacks when sharing connections across sessions.
Approach & patterns
Brad employs a methodical architecture approach with clear separation between essential functionality and enhancement features. The development pattern focuses on getting core IMAP operations working first, then layering on advanced features like threading and real-time updates.
Technical patterns include optimistic updates for operations like marking messages as read (updating local cache immediately for instant UI feedback, then handling server responses asynchronously), using MODSEQ-based synchronization for efficient change detection, and implementing hybrid approaches that combine multiple strategies for robustness (like avatar detection using multiple fallback methods).
The codebase emphasizes minimal dependencies and clean separation of concerns, with separate Node.js threads per active user for database operations to keep the main thread responsive.
Tools & resources
Tech stack: Node.js backend with @toptensoftware/sqlite, codeonlyjs frontend SPA. Dovecot IMAP only. Docker deployment. Brad prefers markdown for email composition, simple folders (inbox/sent/archive/junk), search-focused workflow, keyboard shortcuts essential. Hitch uses separate Node.js thread per active user for database ops, keeping main thread free. SQLite DB per user. Hitch: WebSocket for push updates. IMAP IDLE only on inbox. SQLite stores minimal metadata for threading, actual email content from IMAP. FTS via Dovecot.
Key libraries include ImapFlow for IMAP operations (with custom extensions for SORT functionality), crypto module for secure session management, and various IMAP extensions like QRESYNC, CONDSTORE, and Dovecot-specific features like SNIPPET=FUZZY and PREVIEW=FUZZY for message previews.
Development environment uses docker-mailserver for local testing, with WSL integration for development workflow. The project targets deployment as Docker containers with proper security considerations for password storage and session management.
